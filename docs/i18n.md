# i18n Guide

Vertext 0.0.7 introduces a lightweight i18n foundation: a translation file structure, the `App\CMS\I18n` class, a global `__()` helper, an admin locale switcher, and a `lang` column on content tables for future use.

> **Scope** - This is a foundation release. Translation management UI and URL path-prefix routing are planned for 0.0.8. For now, the locale controls the admin UI language and the date formatting locale.

---

## File Structure

Translation files live in `App/Lang/{locale}/`:

```
App/Lang/
├── en/
│   ├── app.php       # Front-end strings (nav, buttons, error pages)
│   └── admin.php     # Admin UI strings (save, delete, status labels, etc.)
└── id/
    ├── app.php       # Indonesian stubs
    └── admin.php     # Indonesian stubs
```

Each file returns a plain associative array:

```php
<?php
// App/Lang/en/admin.php
return [
    'save_changes' => 'Save Changes',
    'record_created' => ':name created successfully.',
];
```

---

## Using `__()`

The global `__()` helper resolves a dot-notation key against the current locale:

```php
echo __('admin.save_changes');            // "Save Changes"
echo __('app.nav_home');                  // "Home"
echo __('admin.record_created', ['name' => 'Post']); // "Post created successfully."
```

**Key format:** `{file}.{key}`

- `admin.save` - loads `App/Lang/{locale}/admin.php`, looks up `save`
- `app.nav_home` - loads `App/Lang/{locale}/app.php`, looks up `nav_home`
- Keys without a dot look in `app.php`

If a key is missing in the current locale, the English fallback is tried. If that also misses, the raw key string is returned.

## Locale Resolution

`I18n::getLocale()` returns the active locale using this priority:

1. `$_SESSION['vtx_locale']` - set when the user switches locale explicitly
2. `default_locale` in the `settings` table - site-wide default (seeded as `'en'` by `I18n::migrate()`)
3. `'en'` - hard fallback

Locales are discovered by scanning `App/Lang/` for subdirectories. Adding a new locale directory makes it immediately available to `getSupportedLocales()`.

---

## Admin Locale Switcher

When more than one locale is present in `App/Lang/`, a compact language `<select>` appears in the admin top bar between the theme toggle and the user menu.

Selecting a new locale submits a form to `POST /admin/settings/set-locale`, which calls `I18n::setLocale()` (stores locale in session) and redirects back to the current page.

---

## Front-End Locale Switching

Append `?lang={locale}` to any front-end URL to switch the session locale for that visitor:

```
https://example.com/?lang=id
```

Vertext reads `$_GET['lang']` at the top of `Config/Routes.php` before any routing and calls `I18n::setLocale()`.

---

## Date Formatting

```php
echo I18n::date(time(), 'medium');    // "30 Jun 2026"
echo I18n::date('2026-01-15', 'full'); // "Thursday, 15 January 2026"
```

Format options: `'short'`, `'medium'` (default), `'long'`, `'full'`.

When PHP's `ext/intl` extension is available, `IntlDateFormatter` is used for locale-correct output (e.g. Indonesian month names in `id` locale). Without `ext/intl`, a fixed `date()` format map is used.

---

## Adding a New Locale

1. Create the directory: `App/Lang/fr/`
2. Add at minimum `app.php` and `admin.php` returning translation arrays.
3. The locale (`fr`) immediately appears in `getSupportedLocales()` and the admin switcher.

---

## Schema Migration

The `lang VARCHAR(10) NOT NULL DEFAULT 'en'` column is included in the `posts` and `pages` CREATE TABLE statements as of Vertext 0.0.7.

For existing installs (tables created before this version):

1. Go to **Admin - Settings**.
2. Click **Run Migration**.

This calls `I18n::migrate()`, which:
- Adds `lang` to `posts` and `pages` if not already present (using `IF NOT EXISTS`).
- Inserts a `default_locale` row in the `settings` table.

The operation is idempotent and safe to run multiple times.

## Planned for 0.0.8

- **Translation management UI** - edit translation strings from the admin panel without touching files.
- **URL path-prefix routing** - `/id/blog/...` serves Indonesian content; language negotiated from URL prefix, session, or `Accept-Language` header.
- **Content filtering** - public page/post queries will `WHERE lang = '{locale}'` with English fallback, so you can publish separate content per language.
